Overview:

We seek to develop a general reader application for publications on Princeton’s campus. This system would be built to aggregate all of the major publications on campus (The Daily Princetonian, Nassau Weekly, etc.) and publish them to a mobile platform. This system would be general enough to easily accommodate more publications as they form as well as to accommodate changes within publications over time. Our first test case is to build a mobile application for the Daily Princetonian, called, “The Prowler.” We have approached the Daily Princetonian and told them that we would like to make a mobile reader application for their publication and they are very excited. They gave us suggestions about features that they would like it to have. We hope to enhance the aesthetic and functional experiences of users as well as to provide the Daily Princetonian with a way to engage the student body directly. The application would be a way for the Daily Princetonian to publish to a mobile platform. Enhanced features such as archival search, commenting, and a personalized notification system would allow the Daily Princetonian to engage in more meaningful ways with its reader base.

Requirements and Target Audience:

Currently there is no general reader or centralized application for accessing Princeton publications. Like other publications, the Daily Princetonian’s website is currently accessible via mobile devices, however it is aesthetically displeasing as well as limited in functionality (there’s only commenting functionality with little engagement). Since it is hard to read the Daily Princetonian on mobile devices, it makes it difficult to access this publication on the go. The Prowler seeks to make the Daily Princetonian (as well as other publications) more accessible. We would also like to extend the functionality to include a recommendation system, better comments, and archival search. Our basic structure builds off of application systems like Google News etc. A baseline would provide a scrolling feed of headlines which then would then expand into short descriptions. Individual articles would have pages designated to them. Fancier implementations would organize by section, preference, etc.

Functionality:

Use case: quick news updates and personalized notifications: Ron receives a notification from the app saying that there is a new article about basketball. Ron opens the app and it takes him to the article where he reads it but mostly looks at the pictures. He clicks the home page button, which brings him to a personal curation of recommended articles. He skims through them quickly and although none of them catch his attention today, he gets enough of each article to get a general idea of the happenings on campus. He stars an article about a deer infiltrating Forbes. That will be a good read for later. He also stars an article about a policy change. After all, he has smart friends he needs to impress so he doesn’t seem out of touch with news.

Use case 2: searching through historical archives: Ron’s smart friend Hermione is not interested in daily news, since she seeks out papers with a greater scope than campus. However, she loves history and digging into old archives. Hermione opens the app and searches for the word “trend” to see where the earliest occurrence of the word is in publication. She clicks the drop-down menu and clicks order by “date: oldest to newest” and finds her desired article. She wants to potentially do further research on this, so she sends the article to herself.

Use case 3: sharing articles through social media: Mr. Lucius Malfoy is an alum of the school, but he simply cannot forget the great times he had here at Princeton. Therefore, he consistently follows up on the happenings through campus news, but he is also wants people to hear what he has to say. He opens up the app, and immediately clicks the drop-down menu and clicks order by “most read”, and clicks on the first link. He then shares it on Facebook so his 500 friends will see it, and writes a lengthy comment on its implications.

Design:

Our implementation consists of the three standard interfaces: a user interface, a web service, and a database. The user interface will be an IOS application, the web service will be written in python, and the database is a standard MySQL database. This general framework makes “The Prowler” extensible to other publications (not just the Prince) as our UI can abstract each publication. We would make new database tables (via python scripts for each new publication). The only portion that would have to be hard coded are the web scrapers.

UI: The user interface has to be dynamic as the data being supplied to it will change daily - also if we add publications, it should be robust enough to accept that change. Our baseline is a scrolling feed with articles, the next step is a main page organized by section, and then a fully flushed recommendation system (with user login). Emily (emt4) and Eric (eshayes) are tackling this portion of the project.

Web Service: Each communication with the database will be handled through a web service python script running on an online server (we are using Heroku to test, the Daily Princetonian has agreed to pay for formal service for the final implementation). These requests will be handled with an HTTP protocol and JSON format data. The webservice will supply the UI with daily data, so it will support high level operations in stages: GET_ARTICLE, GET_SECTION, and finally GET_PUBLICATION. So, there will be a component of the web service which communicates only with the IOS-Application, we call this the “ios-service.” Another portion of the web service will be devoted to scraping each publication for article content, title, author, date and then supplying this scraped content in JSON format to the MySQL database. We call this the “scraping-service.” We have almost finished implementing the scraping-service for the Daily Princetonian, i.e. we have finished abstracting scraping for articles by URL. We are currently having trouble making the HTTP request work without connection exceptions. Fabian (folr) and Aamir (aamirz) are handling this portion.

Database: The publication data will be stored consistently between sessions in a MySQL database which we can use with Heroku but also locally for testing. There is a table for publication data (articles), one for user data, and one for user metadata associated with each article (comments etc.). Fabian (folr) is working on this portion and already has set up some tables, we are just working on the web service component now to populate the database.
